# OCI FortiGate VM Reshape Automation

Bash scripts to automate the reshaping of FortiGate Active/Passive HA virtual machines on Oracle Cloud Infrastructure (OCI).

Oracle Cloud does not allow changing the shape (VM type) of an instance that has more than one secondary VNIC attached. These scripts handle the full workflow automatically:

1. Fetch current VNIC and volume configuration
2. Detach secondary VNICs
3. Reshape the VM to the target shape
4. Reattach all VNICs restoring the original IPs, names, subnets, and network settings

## Network Interfaces

Each FortiGate VM has 4 VNICs. The primary (mgmt) is never touched. The 3 secondary VNICs are detached and reattached in the following order:

| Order | Interface | Role |
|---|---|---|
| Primary | mgmt | Management â€” **never detached** |
| 1st | public | WAN / Untrust |
| 2nd | trust | LAN / Internal |
| 3rd | hasync | HA Synchronization |

## How It Works

### Detach (Step 2)

The script queries OCI **live** for current attachment IDs â€” it never relies on the backup file for detach operations, which avoids failures due to stale IDs. Each VNIC is detached using:

```bash
oci compute vnic-attachment delete --vnic-attachment-id <live_id> --force
```

The script polls OCI every 15 seconds until the attachment is confirmed removed, with a settling period between each VNIC.

### Reshape (Step 3)

```bash
oci compute instance update \
  --shape "VM.Standard.E5.Flex" \
  --shape-config '{"ocpus": 4, "memoryInGBs": 64}'
```

If OCI returns a `409 Conflict` (instance still processing), the script automatically retries up to 10 times with a 30-second wait between attempts.

### Reattach (Step 4)

VNICs are reattached using the backup JSON as the source of truth for:

- Private IP address
- Subnet
- Display name
- NIC index
- Skip source/destination check
- Hostname label
- NSG assignments

The new attachment ID is resolved by comparing the VNIC attachment list before and after the attach call, avoiding reliance on CLI response parsing.

If a VNIC IP is already allocated (e.g. from a previous partial run), the script detects the conflict and skips that VNIC safely.

## Troubleshooting

**`Error: No such command 'detach'`**
The script uses `oci compute vnic-attachment delete`. If you see this, you may be running an older version of the script. Download the latest version.

**`409 Conflict â€” instance is currently being modified`**
OCI is still processing a previous operation. The script retries automatically. If it persists after 10 retries, wait 2â€“3 minutes and re-run.

**`IP address has already been allocated`**
The VNIC was already successfully attached in a previous run. The script skips it and continues â€” this is safe.

**VNIC is shown as attached in OCI console but script reports detached**
The script uses live OCI data for detach operations. If the console shows the VNIC as attached but the detach appears to succeed, wait 30 seconds and check the console again â€” OCI propagation can be delayed.

**FortiGate not responding after reshape**
This can be caused by interface remapping inside FortiOS after the VM hardware changes. Connect via the OCI serial console (OCI Console â†’ Instance â†’ Console Connection) and verify port-to-IP mappings with:
```
get system interface physical
```

## Important Notes

- The **primary (mgmt) VNIC is never detached** â€” it remains attached throughout the entire process
